# Coze 飞书实施方案（杭电龙虾 MVP）

## 1. 项目目标

先不追求“一次做全”，而是先落地一个能被同学真实使用的最小闭环版本：

- 在 **Coze** 中搭建一个可配置、可迭代的校园助手工作流
- 在 **飞书** 中提供统一入口，承接消息、卡片、提醒和人工确认
- 优先验证三类核心价值：
  - **信息收集总结**：把校园通知、活动、二课、志愿信息做聚合和摘要
  - **通知提醒**：按用户订阅和时间规则主动推送
  - **轻执行闭环**：先做“人工确认后执行”的任务型能力，而不是一开始就全自动

一句话定位：

**先做一个接入飞书的校园 AI 秘书，再逐步进化成多 Agent 的“杭电龙虾”。**

---

## 2. 为什么先选 Coze + 飞书

### 2.1 Coze 的价值

- 适合快速搭建 **工作流、知识库、插件调用、定时任务**
- 能先把复杂逻辑可视化，降低前期开发成本
- 方便快速试错：Prompt、节点编排、工具调用都能较快迭代
- 适合做 MVP 验证，等流程稳定后再考虑迁移到 LangGraph / 自建后端

### 2.2 飞书的价值

- 适合作为统一触达入口：聊天、机器人、卡片消息、审批/确认
- 对“任务待确认”“预约结果通知”“订阅提醒”这类场景很合适
- 团队协作与灰度测试方便，适合先在校内小范围跑起来

### 2.3 当前阶段不建议一开始就重投入的部分

- 不建议第一期就做完整多 Agent 编排平台
- 不建议第一期就打通所有校园系统自动执行
- 不建议第一期就做复杂游戏化体系和 XP 系统

这些都适合放在 MVP 跑通之后逐步追加。

---

## 3. 第一阶段产品边界（MVP）

第一阶段只做 3 个能力域。

### 3.1 信息秘书

目标：解决“信息散、更新慢、找不到”的问题。

功能范围：

- 聚合学校公开通知
- 聚合二课/志愿/讲座/比赛等信息
- 按主题生成摘要
- 支持用户问答检索
- 支持“最近有什么”和“帮我总结今天/本周更新”

数据来源建议优先级：

1. 学校官网、学院官网、教务处、学工相关页面
2. 二课、讲座、志愿等公开页面
3. RSS 或其他公开订阅源
4. QQ 群、校园墙、频道类信息

其中第 4 类建议第一期谨慎处理，因为：

- 数据稳定性差
- 合规和权限问题更复杂
- 内容噪声高，清洗成本大

### 3.2 通知秘书

目标：让用户不用反复来问，系统主动提醒。

功能范围：

- 明日天气提醒
- 明日课程提醒
- 新活动/新通知提醒
- 按兴趣订阅提醒
- 每日/每周摘要推送

推送形态：

- 飞书机器人私聊推送
- 飞书群推送（适合测试阶段）
- 飞书卡片消息，附带“查看详情 / 取消订阅 / 稍后提醒”

### 3.3 执行秘书（轻量版）

目标：先验证“任务代理”体验，但不一开始做高风险全自动。

第一期建议只做：

- 任务登记：例如“帮我盯图书馆座位”
- 条件监控：例如“有空位时提醒我”
- 人工确认：飞书卡片确认“是否继续执行”

第一期先不建议直接做：

- 自动代登录校园系统
- 自动提交请假
- 自动抢馆/抢场地

原因：

- 风险高
- 依赖复杂鉴权
- 容易引发封禁、账号安全和责任界定问题

## 4. 推荐实施路线

建议分三期做。

### Phase 1：跑通可用闭环

目标：让用户在飞书里能真实使用。

交付内容：

- Coze Bot 1 个
- Coze 工作流 2 到 4 条
- 飞书机器人 1 个
- 基础知识库 1 套
- 定时推送能力
- 人工确认卡片能力

产出结果：

- 用户可以提问校园信息
- 用户可以订阅提醒
- 用户可以收到结构化通知
- 用户可以发起“帮我盯一下”的任务

### Phase 2：增强任务能力

目标：让“提醒”变成“半自动执行”。

交付方向：

- 增加更多任务模板
- 引入用户偏好和个人画像
- 增加任务状态管理
- 对接更多外部工具或中间服务

### Phase 3：演进为多 Agent 龙虾

目标：把“秘书”升级为“多角色协作系统”。

演进方向：

- 信息 Agent
- 通知 Agent
- 执行 Agent
- 游戏化 Agent
- 调度 Agent / 裁判 Agent

这一阶段再考虑是否迁移到 LangGraph 或自建调度框架。

## 5. Coze 工作流设计建议

建议不要第一天就做一个超大总工作流，而是拆成多个单用途流程。

### 5.1 工作流 A：校园信息问答

输入：

- 用户问题
- 用户身份标签（年级、学院、兴趣，可为空）

流程：

1. 识别问题类型
2. 检索知识库
3. 必要时调用搜索/抓取插件
4. 汇总答案
5. 返回飞书

适用问题：

- 最近有什么志愿活动
- 明天有什么讲座
- 帮我总结本周通知

### 5.2 工作流 B：订阅与提醒管理

输入：

- 用户自然语言指令

流程：

1. 解析订阅意图
2. 提取订阅对象、频率、渠道
3. 写入用户订阅表
4. 返回确认消息

适用指令：

- 每天晚上 9 点提醒我看明天课表
- 有新的二课活动就告诉我
- 每周给我发一次招聘信息汇总

### 5.3 工作流 C：定时巡检与推送

触发方式：

- Coze 定时任务 / 外部调度器

流程：

1. 拉取订阅规则
2. 抓取最新数据
3. 规则匹配
4. 生成摘要
5. 通过飞书推送

### 5.4 工作流 D：任务监控与人工确认

输入：

- 用户发起的任务

流程：

1. 解析任务目标
2. 建立监控任务
3. 定时检查条件
4. 条件满足后发送飞书卡片
5. 用户确认后再执行后续动作或结束任务

这个流程很适合你文档里提到的“类似候补”和“人工确认”思路。

## 6. 飞书接入设计建议

### 6.1 飞书里承接什么

飞书主要负责四件事：

- 作为统一聊天入口
- 展示结构化卡片
- 接收用户确认操作
- 承接主动推送

### 6.2 推荐的飞书交互形式

1. 机器人私聊
   适合个人提醒、任务确认、订阅管理

2. 群机器人
   适合测试期的信息播报、版本联调、运营观察

3. 卡片消息
   适合以下场景：
   - 预约结果通知
   - 是否确认执行
   - 订阅成功/变更
   - 查看详情和快捷按钮

### 6.3 飞书侧最小能力清单

- 接收用户消息
- 把消息转发给 Coze
- 接收 Coze 返回内容
- 发送普通文本/卡片消息
- 处理卡片按钮回调

---

## 7. 数据与知识库设计

### 7.1 第一阶段建议的数据分层

建议拆成三层：

1. **公共知识层**
   学校通知、办事流程、公开活动、FAQ

2. **动态资讯层**
   最近更新的活动、讲座、比赛、招聘、志愿信息

3. **用户订阅层**
   用户订阅规则、提醒偏好、任务记录

### 7.2 知识更新方式

- 静态知识：手动导入 + 定期更新
- 动态资讯：定时抓取 + 清洗入库
- 用户数据：通过工作流结构化存储

### 7.3 第一阶段不必过度复杂化

MVP 阶段可以先不做特别重的向量架构，建议：

- 公共 FAQ 和规则说明：结构化知识为主
- 动态通知与活动：列表 + 标签 + 摘要
- 复杂文本检索：再逐步引入向量检索

也就是说，第一期不是“为了 RAG 而 RAG”，而是先保证信息真的准、能更新、能推送。

## 8. 角色拆分建议

虽然最终目标是多 Agent，但第一期建议逻辑上拆角色，技术上不必真的做成多个复杂自治 Agent。

建议先抽象为 4 个角色：

- **路由器**：识别用户是来问信息、订阅提醒、还是发起任务
- **信息秘书**：负责检索、总结、回答
- **通知秘书**：负责订阅、生成提醒、主动推送
- **执行秘书**：负责监控型任务和确认型任务

这会让后面迁移到 LangGraph 更顺，不会把逻辑全揉在一个 Bot 里。

## 9. 风险与难点

### 9.1 校园系统执行风险

自动代操作类能力最大风险在这里：

- 账号密码管理
- 登录态维护
- 验证码和风控
- 责任归属

所以第一期建议全部采用：

**监控提醒 + 人工确认 + 半自动执行**

### 9.2 数据来源合规性

尤其是 QQ 群、校园墙、频道信息：

- 可能存在隐私问题
- 可能存在抓取限制
- 内容真实性不稳定

建议优先做公开来源，再逐步引入人工审核或白名单源。

### 9.3 Coze 平台边界

Coze 很适合 MVP，但可能遇到：

- 长周期任务状态管理受限
- 复杂多 Agent 调度能力有限
- 外部系统深度集成时灵活度不足

所以策略建议是：

**先用 Coze 验证产品，后续把重状态、重执行、重调度能力逐渐外移。**

## 10. 你现在最适合先做的版本

如果以“尽快跑起来”为目标，我建议你先做下面这个版本：

### MVP v0.1

- 飞书机器人作为唯一入口
- Coze 里建 3 条工作流：
  - 校园信息问答
  - 订阅管理
  - 定时推送
- 只接公开数据源
- 先不上自动执行
- 先不上游戏化

### MVP v0.2

- 增加“帮我盯一下”任务流
- 飞书卡片人工确认
- 增加简单用户画像

### MVP v0.3

- 引入任务中心
- 引入 XP/等级展示
- 逐步拆成真正的多 Agent 结构

---

## 11. 建议的本周推进清单

如果我们这周就开始推进，建议按这个顺序：

1. 明确第一期只做哪 3 个场景
2. 列出第一批数据源清单
3. 画出 Coze 工作流草图
4. 确定飞书机器人交互方式
5. 先联通“问答 + 推送”两个最小闭环
6. 再决定要不要接“监控型任务”

---

## 12. 我对这个项目的判断

这个方向是成立的，而且很适合先用 Coze 验证。

真正的关键不在于“模型多强”，而在于三件事：

- 信息源是不是稳定
- 工作流是不是清晰
- 飞书里的交互是不是足够顺手

只要先把“问得到、推得到、点得动”做顺，杭电龙虾这个项目就已经有很强的展示价值和继续扩展的基础了。
